# 录音识别同步短音频模型的体积压缩方案（Opus / MP3）

## 背景

`fun-asr-flash-2026-06-15` / `qwen3-asr-flash` 走同步 `multimodal-generation` 接口，音频要以
`data:<mime>;base64,<data>` 的 Data URI 形式内嵌到请求体里。文档明确写着"Base64 编码会增大体积，
请控制原文件大小，确保编码后仍符合输入音频大小限制（10MB）"——即限制是**对编码后的体积**生效的，
原始文件在 base64 膨胀（约 ×4/3）后必须仍 ≤10MB，也就是原始文件本身要控制在 7.5MB 左右。

只做"降采样到 16kHz + 下混单声道"（阿里云文档自己给的 ffmpeg 处理建议）并不够：一段接近上限的
5 分钟录音，16kHz/16-bit/单声道 PCM WAV 约 9.15MB，base64 后 ~12.2MB，仍会超限。所以还需要做一次
有损压缩。

## 两个模型不能用同一种压缩格式

- **fun-asr-flash-2026-06-15**：请求体里有独立的 `parameters.format` 字段，文档（`docs/API/阿里云百炼/Fun-ASR录音文件识别HTTP API参考.md` 第 211 行）明确写着支持 `wav`、`mp3`、`opus`。可以放心用 Opus。
- **qwen3-asr-flash**：请求体没有 `format` 字段，音频格式完全靠 Data URI 的 `mediatype`
  （`data:<mime>;base64,...`）判断，文档里只出现过 `audio/wav`、`audio/mpeg` 两种验证过的示例，
  没有任何地方证实它认识 `audio/opus`。给它用 Opus 就是重蹈"凭猜测传未验证格式"的覆辙（参见
  [[录音识别短音频模型不能复用OSS临时上传地址]]那次的教训），所以这个模型改用 MP3。

## 实现

- `src-tauri/src/audio_prep.rs`：用 `symphonia`（0.6 版本的新 API，`get_probe().probe(...)` +
  `default_track(TrackType::Audio)` + `make_audio_decoder(...)`，不是旧版的 `SampleBuffer`/
  `CODEC_TYPE_NULL` 那一套）解码任意输入文件的音轨，下混单声道、重采样到 16kHz（复用
  `audio_dsp.rs` 里泛化出来的 `resample_linear`，原来是给实时降噪管线用的私有 `resample_to_48k`）。
- `src-tauri/src/providers/alibabacloud/transcription.rs`：
  - `build_opus_data_uri`：Opus 编码用 `ogg-opus` crate（`ogg_opus::encode::<16000, 1>(&pcm16)`），
    这个库正是为 16kHz/单声道场景写的，底层绑定 `audiopus`/libopus（BSD-3，无 GPL/LGPL 问题）。
  - `build_mp3_data_uri`：MP3 编码用 `mp3lame-encoder` crate（LGPL-3.0，绑定 LAME）。
- 两个函数返回值：Opus 用 `audio/ogg` mime；MP3 用 `audio/mpeg` mime（跟文档里验证过的示例一致）。

## 压缩效果（5 秒 44.1kHz 立体声测试音频，见 `transcription.rs` 里的单元测试）

- 原始 PCM：882,000 字节
- MP3（mono/16kHz/64kbps）：40,032 字节（约 1/22）
- Opus（mono/16kHz）：15,709 字节（约 1/56）

按比例换算到 5 分钟音频，MP3 约 2.4MB、Opus 约 0.9MB，距离 10MB 上限有充分余量。

## 依赖与许可注意

- `symphonia`：MPL-2.0。
- `ogg-opus` → `audiopus` → libopus：MIT / BSD-3，无 copyleft 负担。
- `mp3lame-encoder` → LAME：**LGPL-3.0**，这是本次唯一引入的 copyleft 依赖，用户已确认接受。
  如果将来要发布/分发这个应用，需要留意 LGPL 对闭源应用静态链接的合规要求（提供许可文本、
  可重新链接说明等），不要当作普通宽松许可对待。

## 适用场景

以后如果要给"同步/一次性返回结果"的语音识别模型接入新的压缩格式，先确认该模型的请求协议里，
音频格式是由**独立字段**（如 `format`）指定，还是只能靠 **Data URI 的 mediatype** 推断——
后一种情况下只能用文档里已经给出示例验证过的 mime 类型，不能凭空替换成"应该支持"的格式。
