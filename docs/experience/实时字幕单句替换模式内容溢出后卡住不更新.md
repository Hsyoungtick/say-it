# 实时字幕「单句替换」模式：一句话说长了之后悬浮窗看起来卡住不更新

## 现象

设置为「单句替换」（`subtitle.displayMode === "replace"`，固定只显示 1 行）后，
一句话说得比较长、文字宽度超过一行时，悬浮窗看起来从某个时刻起就不再更新了，
后面说的内容像是没有显示出来。

## 排查过程中的误判

一度怀疑是 `requestAnimationFrame` 被节流导致的（指示器窗口创建时 `focused(false)`，
WebView2 可能把非焦点窗口的 rAF 降频甚至暂停）。为此把 `IndicatorApp.tsx` 里
`renderText` 从"rAF 排队绘制"改成"收到事件就同步绘制"（commit `8f7df25`），
但很快被自己 revert 掉（commit `44a01e3`）——因为这并不是根因，字幕文字状态其实
一直在正常更新，只是没画出来对用户不成立。

## 真正的根因

`applyScroll()`（`ui/src/indicator/IndicatorApp.tsx`）里，`replace` 模式被特殊处理成
**永远强制 `translate3d(0, 0, 0)`，不做任何滚动**：

```ts
if (mode === "subtitle" && subtitleConfig.displayMode === "replace") {
  const next = "translate3d(0, 0, 0)";
  ...
  return; // 不执行下面的溢出滚动逻辑
}
```

而 `syncSubtitleIndicator()`（`ui/src/features/subtitles/controller.ts`）里，
`replace` 模式的框高固定按 `effectiveLines = 1` 计算，不受"显示行数"设置影响。

问题就出在这两者叠加：`#text-content` 用 `white-space: pre-wrap` 换行，一旦当前这句话
渲染宽度超过一行，多出来的文字会自然换到第二行、第三行，但 `#text-flow` /`#text`
都是 `overflow: hidden` 且只留 1 行高度。由于 `applyScroll` 在 replace 模式下把
transform 锁死在 `(0,0,0)`，视图永远只显示"第一行"，超出的后续内容被裁在看不见的
下方——**DOM 里的文字其实一直在正确更新**，只是可见区域没有跟着滚动上去，看起来就像
"填满方框后不再更新"。

## 修复

去掉 `applyScroll` 里 replace 模式的特殊分支，统一用原有的"按溢出量向上滚动"逻辑
（`overflow = scrollHeight - flow.clientHeight`，`offset = max(overflow, 0)`）。
内容没溢出时 offset 自然是 0，效果和之前一致；一旦当前这句话换行溢出单行框，
会像 `scroll` 模式一样把视图滚到能看到最新文字的位置，不会再卡住。

## 教训 / 可复用经验

- 遇到"界面看起来卡住不更新"类问题，不要默认是"状态没更新"或"渲染时机被节流"，
  先确认 DOM/state 是否真的在变化（可以先加日志或临时去掉 CSS `overflow: hidden`
  验证），再决定是状态问题还是纯视觉裁剪/滚动问题。
- 涉及固定高度 + `overflow: hidden` 的展示框，只要内容可能超出固定行数，就必须有
  滚动或截断策略；"永远锁定在顶部不滚动"这类特殊分支要谨慎，容易在内容变长时
  制造出"看起来没更新"的假象。
