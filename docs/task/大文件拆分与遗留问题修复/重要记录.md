# 重要记录

## 记录 001：`providerKind` 字段改名为 `providerId` 而非删除

- 日期：2026-07-06
- 状态：已确定
- 背景：全仓库审查发现注册表字段 `providerKind` 存的是 provider id（`"funasr"`），与 profile 的 `kind`（`"alibabacloud-funasr"`）语义冲突，且当前前后端均无逻辑消费方。可选方案：a) 删除字段；b) 改名 `providerId`。
- 最终决定：改名为 `providerId`，保留字段。
- 选择原因：接入第二家供应商后需要按供应商过滤模型（模型属于哪个 profile），删除会丢失将来必需的信息；改名成本极低（数据文件 + 两处解析结构 + 手册），且用户已批准修复这批问题。
- 影响任务：1.1。

## 记录 002：`subtitles` 场景采用「补消费方」而非「删场景值」

- 日期：2026-07-06
- 状态：已确定
- 背景：注册表 `scenes` 声明了 `subtitles` 场景，操作手册承诺 `scenes` 决定模型出现在哪些下拉，但字幕面板实际复用 `dictationRealtime` 场景的选项，`subtitles` 无消费方。可选方案：a) 字幕面板改为消费 `optionsForScene("subtitles")`；b) 从注册表和手册中删掉该场景值。
- 最终决定：方案 a，补消费方。
- 选择原因：手册与注册表数据都已按「字幕是独立场景」设计，补消费方是解决根因且保留表达能力（未来可上线只适合字幕或只适合听写的模型）；删场景值反而缩小契约。当前数据下两场景模型集合一致，下拉内容不变，无行为风险。
- 影响任务：1.1。

## 记录 003：拆分任务的统一红线——只搬运不改逻辑

- 日期：2026-07-06
- 状态：已确定
- 背景：2.1、2.2、2.3、3.1、3.2 五个拆分任务涉及听写、字幕、转写、对齐等核心链路，且仓库无自动化 UI 测试，回归主要靠 `cargo test` + 用户实测。
- 最终决定：所有拆分任务只允许三类代码变化——模块/import 声明、可见性修饰、任务文件中明确列出的结构性变化（2.3 的闭包提取为具名函数、3.1 的会话状态收敛为单对象、类型别名）；函数体一律逐字搬运，发现原代码可优化也不顺手改，另行记录。
- 选择原因：拆分与逻辑修改混在同一提交会让回归定位失效；行为不变才能用「同输入同输出」验收。
- 影响任务：2.1、2.2、2.3、3.1、3.2。
