# 大文件拆分与遗留问题修复 · 实施方案

## 项目现状速览（供无历史对话的执行者定位）

- 项目：桌面端语音/听写应用「说吧！」，Tauri 2 + Rust 2021 后端，React 19 + TypeScript + Vite 6 前端，主打包目标 Windows。
- 后端代码在 `src-tauri/src`：`commands/` 为 Tauri 命令层，`providers/` 为供应商接入（注册表 `registry.rs`、连接器 `connector.rs`、阿里云实现 `alibabacloud/`），`desktop/` 为桌面能力，根级有 `text_align.rs`（文稿对齐算法）、`audio_dsp.rs`、`audio_prep.rs` 等。
- 前端代码在 `ui/src`：`views/`（页面）、`components/`（UI 与业务组件）、`features/`（业务 controller）、`store/`（Zustand）、`lib/`。两个入口：`ui/index.html` 主窗口、`ui/indicator.html` 悬浮窗。
- 模型注册表：仓库根 `shared/asr-models.json`，后端经 `registry.rs` 的 `include_str!` 编译期内嵌，前端经 `~shared` 别名（`vite.config.ts` / `tsconfig.json`）打包内联，是前后端共享数据契约。
- 相关背景任务：`docs/task/供应商与模型扩展性改造/`（已完成），新增模型/供应商流程见 `docs/rules/新增供应商与模型操作手册.md`。
- 验证命令：`cd src-tauri && cargo check && cargo test`（当前 53 个测试）、`npx tsc --noEmit`、`npm run ui:build`。无 lint/format/自动化 UI 测试。

## 当前存在的问题

### 遗留小问题（2026-07-06 全仓库审查结论）

1. **注册表 `subtitles` 场景无消费方**：`shared/asr-models.json` 每个实时模型都声明 `scenes: ["dictationRealtime", "subtitles"]`，操作手册也承诺 `scenes` 决定模型出现在哪些下拉；但全仓库没有 `optionsForScene("subtitles")` 调用，字幕面板 `ui/src/views/RealtimeSubtitlesPanel.tsx:165` 实际使用的是 `REALTIME_ASR_MODEL_OPTIONS`（派生自 `dictationRealtime` 场景）。当前数据下两场景内容一致所以无表现差异，但契约已经失真。
2. **`providerKind` 字段命名误导**：注册表字段 `providerKind` 实际存的是 provider id（`"funasr"`），而 profile 的 `kind` 是 `"alibabacloud-funasr"`；该字段目前前后端都无消费方，新增供应商时极易误导。
3. **死代码**：`src-tauri/src/commands/asr.rs:494-495` 有两个行为完全相同的 `AsrEvent::TaskFinished` match 分支（guard 无效）。
4. **单供应商硬编码文案残留**：静音测试错误前缀写死 `"Fun-ASR 上游错误"`（`commands/asr.rs:497`）；文件听写报错写死 `"请先在设置中保存阿里云百炼 API Key"`（`ui/src/features/dictation/controller.ts:229`）。
5. **`SettingsProviderPanel` 取 profile 方式不一致**：`ui/src/views/SettingsProviderPanel.tsx:58` 取「第一个 enabled 的 asr profile」而非生效供应商（`effective("asr")` / `defaults.asr`）；`normalize_settings` 现已保留未知 profile，极端情况下面板可能渲染错对象并把 API Key 写入错误 profile。

## 最终目标

1. 五项遗留小问题全部修复，注册表契约（`scenes`、`providerKind`）与实现、操作手册三者一致。
2. 五个大文件按职责拆为小文件/模块目录，每个新文件职责可一句话说明；对外公共 API（Rust pub 面、TS 导出面、Tauri 命令注册）保持不变或由原路径 re-export，消费方零改动或仅改 import 路径。
3. 全部验证命令通过，涉及交互的功能经用户实测确认无回归。

## 总体解决思路

- **先修后拆**：第一阶段把小问题清理干净，避免拆分时把死代码、错误文案原样搬进新文件；也避免修复与拆分改同一文件产生冲突（2.3、3.1 依赖 1.2）。
- **拆分只做「搬家 + 收敛可见性」**：函数体逐字搬运，不顺手改逻辑、不改算法、不改消息格式；唯一允许的代码变化是模块声明、import/use 路径、可见性修饰符（`pub` → `pub(crate)`/私有，当且仅当无外部消费方）。
- **保持入口稳定**：Rust 侧用模块目录（如 `transcription/mod.rs`）re-export 原有 pub 项，`alibabacloud/mod.rs` 与 `main.rs` 的对外声明不变；前端保持原导出文件路径（或原路径文件转为聚合导出），消费方 import 尽量零改动。
- **测试跟随代码走**：单元测试搬到对应新模块的 `#[cfg(test)]`；拆分前后 `cargo test` 通过数量不得减少（当前 53 个）。

## 兼容与风险控制

- **持久化**：不改 `say-it-state.json` 结构与 `providers/mod.rs` 的 normalize/序列化逻辑。
- **注册表数据契约**：`providerKind` 改名同时更新后端 `registry.rs`、前端 `modelRegistry.ts`、`docs/rules/新增供应商与模型操作手册.md` 三处，防止手册与实现脱节（该字段当前无逻辑消费方，风险低）。
- **交互回归**：听写注入/取消、实时字幕、静音测试、字幕编辑器拖拽等必须由用户实测，任务文件里列出具体操作步骤与验证点。
- **回退原则**：每个任务独立提交；任一拆分导致行为异常且无法当场定位时，`git revert` 该任务提交回到上一个稳定点，不带病继续。
- **同一路径连续失败两次立即停下重新分析**，不堆补丁（见 CLAUDE.md）。
