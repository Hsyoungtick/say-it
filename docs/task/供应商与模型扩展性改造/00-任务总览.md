# 供应商与模型扩展性改造 · 任务总览

## 任务目标

「说吧！」目前所有识别能力都绑定在阿里云百炼（`funasr`）单一供应商上：后端在多处硬编码供应商 ID 与 `kind` 检查，模型能力信息（协议族、热词支持、时间戳支持、可用场景）以字符串前缀判断的形式散落在前后端十余处。本次改造的目标是：**建立"模型注册表 + 供应商连接器"两层机制，使得未来新增一个同族新模型只需改 1 个数据文件，新增一个新供应商只需新增 1 个 provider 模块 + 注册表登记**，并为后续引入 LLM 等新能力类型（如识别结果后处理）预留能力机制。

**属于本次任务**：模型注册表（单一数据源）及前后端散落判断的收敛；实时识别链路的供应商连接器 trait 抽象；解除后端单供应商硬锁（`normalize_settings` / 命令层硬编码）；供应商默认值结构的多能力化（预留 `llm`）；前端供应商层泛化；用一个真实新模型验证流程并沉淀《新增供应商与模型操作手册》。

**不属于本次任务**：实现任何 LLM 后处理功能（仅预留机制）；接入任何新的第三方供应商；重写文件转写（OSS 上传 / 同步短音频 / SSE）的协议抽象（阿里云内部按模型分支保留，仅把判断依据挪入注册表）；UI 视觉改动；热词功能行为变更。

## 重要说明

- **持久化兼容是硬约束**：用户本地已有 `say-it-state.json`（含 `providers.defaults.asr`、funasr profile 的 apiKey / 热词 / 词表 ID）。任何对 `ProviderSettings` / `ProviderDefaults` 结构的改动都必须能无损反序列化旧 JSON（`#[serde(default)]` 兜底），不得导致用户丢 Key 或丢热词。
- **行为不变原则**：第一、二、三阶段是纯重构，听写 / 实时字幕 / 录音识别 / 热词的对外行为不允许变化；每阶段完成后用相同操作路径回归。
- 涉及麦克风、快捷键、注入等真实交互的验证，改完后列出操作步骤交给用户实测，不要自行模拟。
- 执行任何任务前，先阅读本文件、[01-实施方案.md](01-实施方案.md) 和对应任务文件。
