# 3.2 前端供应商层泛化与硬编码清理

## 任务状态

- 状态：待开始
- 所属阶段：第三阶段-多供应商基础设施
- 前置任务：3.1
- 关联任务：4.1
- 是否可以并行：否（依赖 3.1 的 defaults 结构）

## 任务目标

前端供应商层不再假设单供应商：能力类型可扩展（预留 `llm`）、生效供应商判断按 store 数据而非硬编码 `"funasr"`、设置页能按 profile 列表渲染（当前仍只有一个，UI 外观不变）。

## 当前情况

- `ui/src/store/useProviderStore.ts`：
  - `:4` `type ProviderCapability = "asr"` 字面类型；
  - `:22-24` `ProviderDefaults { asr: string }`；
  - `:50-52` `FALLBACK_DEFAULTS = { asr: "funasr" }`、`:117` `effective()` 兜底 `|| "funasr"`；
  - `saveFunasrHotwords` 等热词 action 是供应商专属，保留。
- `ui/src/features/dictation/controller.ts:91-98` `ensureDictationProviderReady()`：硬编码 `profiles.find(p => p.id === "funasr")?.status?.hasApiKey`，真实意图是"当前生效的 ASR 供应商是否已配置好"。
- `ui/src/views/SettingsProviderPanel.tsx`：供应商设置页，当前围绕单一 funasr profile 渲染（API Key、模型参数、热词入口）。
- 后端 3.1 完成后 `defaults` 会多出 `llm` 字段（空串表示未设置），`list_providers` 响应结构其余不变。

## 实施方案

- `ProviderCapability` 改为 `"asr" | "llm"`；`ProviderDefaults` 增加可选 `llm`；`FALLBACK_DEFAULTS` 与 `effective()` 的兜底改为"defaults 里查不到时取第一个具备该能力的 enabled profile"，不再写死 id。
- `ensureDictationProviderReady()` 改为：`useProviderStore.getState().effective("asr")` 取生效 id → 查该 profile 的 `status.hasApiKey`。报错文案保持现状。
- `SettingsProviderPanel.tsx`：检查其取 profile 的方式，若硬编码 `"funasr"` 则改为遍历 `optionsFor("asr")` 渲染（单个时外观与现状一致）；热词面板（`FunAsrHotwordsPanel.tsx`）保持为 funasr 专属板块，仅在该 profile 存在时显示。
- 本任务不新增任何 llm UI，只保证类型与数据通路不被 `"asr"` 字面量锁死。

## 执行步骤

1. 改 `useProviderStore.ts` 类型与兜底逻辑（`effective` / `FALLBACK_DEFAULTS` / `normalize`）。
2. 改 `features/dictation/controller.ts` 的 `ensureDictationProviderReady`。
3. 全局搜索 `"funasr"` 字面量（`ui/src` 内），逐处判断：供应商专属功能（热词命令、面板）保留并注释；通用逻辑改为按 store 派生。
4. 检查 `SettingsProviderPanel.tsx` 渲染来源，按需改为列表驱动（不改视觉）。
5. `npx tsc --noEmit`、`npm run ui:build` 通过。
6. 用户实测：设置页外观与操作不变；未配 Key 时启动非实时听写提示不变；配好 Key 后全流程正常。
