## 实施方案

1. 确认验证模型（重要记录 004）后，严格按预期流程操作：只改 `shared/asr-models.json`（+ 需要热词时 `customization.rs` 的 `VOCABULARY_TARGETS` 一行），不允许碰其他文件；若发现还需要改别处，说明前三阶段有漏，回到对应任务补而不是在本任务打补丁。
2. 端到端验证新模型在听写/字幕中可用。
3. 依据这次真实操作写手册，包含三个场景的步骤清单、涉及文件、验证方法，以及"新增能力类型（如 llm）"的机制说明（defaults 字段 + capabilities 数据 + 对应命令层，指向本计划实施方案）。

## 执行步骤

1. 与用户确认验证模型与是否保留在正式下拉中（若仅作验证，验证后从注册表移除并在执行记录注明）。
2. 在 `shared/asr-models.json` 添加该模型条目（协议族、场景、能力标记按官方文档填写）。
3. 若需热词：`VOCABULARY_TARGETS` 加一行（新前缀），并在设置页重新保存一次热词生成新词表。
4. `cargo check`、`cargo test`、`npx tsc --noEmit`、`npm run ui:build`；`npm run tauri:dev` 下用户实测：新模型出现在预期下拉、完成一次听写注入、一次字幕。
5. 记录本次实际改动的文件清单，与预期流程比对，偏差写入执行记录。
6. 编写 `docs/rules/新增供应商与模型操作手册.md`（场景一：同族新模型；场景二：新协议族；场景三：新供应商；附能力类型扩展说明），步骤以本次真实操作为准。
7. 更新本计划 00-任务总览为完成态。
