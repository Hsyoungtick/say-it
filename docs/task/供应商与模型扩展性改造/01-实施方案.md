# 供应商与模型扩展性改造 · 实施方案

## 当前现状

### 架构概览（与本任务相关的部分）

- 后端 `src-tauri/src/providers/`：目前只有 `alibabacloud` 一个供应商模块，含 `protocol.rs`（实时 WebSocket 消息构造/解析）、`urls.rs`（连接请求构造）、`transcription.rs`（文件转写协议）、`customization.rs`（热词词表）、`uploads.rs`（临时 OSS）。`providers/mod.rs` 定义 `ProviderProfile / ProviderSettings / ProviderDefaults` 与 `normalize_settings` 等。
- 后端 `src-tauri/src/commands/`：`asr.rs`（实时识别会话循环）、`transcription.rs`（文件转写任务）、`funasr.rs`（热词命令）、`session.rs`（供应商设置命令）、`common.rs`（`resolve_provider_id` 等辅助）。
- 前端：`ui/src/features/asr/modelOptions.ts`（模型下拉与能力判断）、`ui/src/store/useProviderStore.ts`（供应商状态）、`ui/src/views/SettingsProviderPanel.tsx`（供应商设置页）。
- 持久化：`src-tauri/src/persistence.rs` 把 `ProviderSettings` 原样序列化进 `say-it-state.json`，读写两侧都过 `normalize_settings`。

### 现有做得对的部分（保留，不推翻）

- `commands/asr.rs` 的会话循环本身是**协议无关**的：只消费 `FunAsrEvent`（Started / Partial / Final / TaskFinished / TaskFailed / Other）抽象事件流，供应商差异已收敛在"连接请求、开始消息、音频消息、finish 消息、消息解析"五个点上。这是 trait 化的天然切面。
- 供应商配置已有 `id / kind / capabilities / config` 的 profile 结构和 `resolve_provider_id` / `has_capability` 能力查询机制，只是被单供应商假设短路了。

### 问题一：模型能力信息散落（字符串前缀判断遍布前后端）

前端 `ui/src/features/asr/modelOptions.ts`：
- 3 张手工维护的下拉清单（`REALTIME_ASR_MODEL_OPTIONS` / `FILE_ASR_MODEL_OPTIONS` / `DICTATION_FILE_ASR_MODEL_OPTIONS`）；
- 9 个布尔函数（`isSupportedRealtimeModel`、`isDictationFileModel`、`isQwenRealtimeModel`、`isQwenShortAudioFileModel`、`isFunAsrFlashFileModel`、`supportsAlignmentTimestamps` 等），消费方遍布 `views/RealtimeSubtitlesPanel.tsx`、`views/DictationShortcutsPanel.tsx`、`views/TranscriptionView.tsx`、`features/dictation/controller.ts`、`features/transcription/controller.ts`、`features/compare/models.ts`、`store/useDictPrefs.ts`、`store/useTranscriptionStore.ts`、`store/useSubtitleStore.ts`。

后端：
- `providers/alibabacloud/protocol.rs:70-76` `realtime_asr_family()` 用 `starts_with("qwen3-asr-flash-realtime")` 分协议族；
- `providers/alibabacloud/protocol.rs:87` 用 `starts_with("fun-asr") || starts_with("paraformer")` 判断是否支持 `vocabulary_id`；
- `providers/alibabacloud/transcription.rs:511-517` `upload_for_model()` 按模型名分 4 路（filetrans / 短音频 / fun-asr-flash / paraformer+fun-asr）；
- `commands/transcription.rs` `uses_async_transcription_task()` 判断走异步 OSS 还是同步短音频；
- `providers/alibabacloud/customization.rs:11-15` `VOCABULARY_TARGETS` 热词目标模型表，注释里明确要求"与 modelOptions.ts 保持一致"——**同一份知识现在靠注释约定人工同步**。

### 问题二：单供应商硬锁

- `providers/mod.rs:124` `normalize_settings` 里 `retain(|p| p.id == FUNASR_PROVIDER_ID)`——任何非 funasr 的 profile 会被直接删除，且 `display_name / capabilities` 被强制覆写；
- `commands/asr.rs:30` `if profile.kind != "alibabacloud-funasr" { return Err(...) }`；
- `commands/transcription.rs:304-315`（`funasr_api_key`）、`320-337`（`funasr_hotword_state`）直接 `find_profile(FUNASR_PROVIDER_ID)`，绕过 `resolve_provider_id`；
- `commands/funasr.rs:23-33` 热词命令硬编码（热词本身是阿里云专属功能，可保留，但取 Key 的方式要与供应商机制一致）；
- `commands/asr.rs:485` `run_asr_silence_test` 拒绝 Qwen 家族，只支持 duplex 协议；
- 前端 `features/dictation/controller.ts:91-98` `ensureDictationProviderReady` 硬编码 `profile.id === "funasr"`；
- 前端 `store/useProviderStore.ts:4` `ProviderCapability = "asr"` 字面类型、`:117` `effective()` 兜底写死 `"funasr"`。

### 问题三：能力类型单一

`ProviderDefaults` 只有 `asr: String` 一个字段；`has_capability` / `set_default_provider` 都写死 `capability == "asr"`。未来引入 LLM 后处理（对识别结果做润色/纠错）时无处安放"默认 LLM 供应商"这类概念。

## 最终目标

改造完成后，固定流程应达到：

| 场景 | 需要改动的文件 |
|---|---|
| 新增同协议族的新模型（如 fun-asr-realtime 新快照版） | 模型注册表 1 处（+ 若支持热词，`VOCABULARY_TARGETS` 1 行） |
| 新增同供应商的新协议族模型 | 注册表 1 行 + 该供应商 provider 模块内新增一个连接器实现 |
| 新增新供应商 | 新建 `src-tauri/src/providers/<供应商>/` 模块 + 内置 profile 与注册表登记 +（若有专属配置项）设置页一个面板 |
| 新增新能力类型（如 llm） | capabilities 数据 + defaults 字段 + 对应的命令层，机制无需重构 |

并有一份《新增供应商与模型操作手册》（`docs/rules/`）描述每种场景的确切步骤。

## 总体方案

### 1. 模型注册表（第一阶段）

建立一份结构化模型元数据，作为唯一数据源。每条记录包含：

```
id（模型名）、显示名、供应商 kind、类别（realtime / file）、
协议族（dashscope-duplex / qwen-realtime / file-async-oss / file-sync-short-funasr / file-sync-short-qwen）、
支持热词 vocabulary_id（布尔）、支持对齐时间戳（布尔）、
可用场景（听写实时 / 听写文件 / 字幕 / 录音识别，位标记或数组）、
是否默认（听写实时默认 / 录音识别默认）
```

载体方案见重要记录 001（推荐：仓库根 `shared/asr-models.json`，Rust 侧 `include_str!` + serde 解析为静态表，TS 侧直接 import，Vite 打包内联；两侧各写一层薄封装函数，散落判断全部改为查表）。

前缀匹配（`starts_with`）保留为注册表的**兜底策略**：用户手输/历史配置里出现注册表外的模型名时，按前缀推断协议族，行为与现状一致。

### 2. 实时连接器 trait（第二阶段）

在 `providers/` 定义：

```rust
pub trait RealtimeAsrConnector: Send {
    fn connect_request(&self) -> Result<Request<()>, String>;
    fn start_messages(&self) -> Vec<Message>;          // run-task / session.update
    fn audio_message(&self, bytes: Vec<u8>) -> Message; // Binary / base64 JSON
    fn finish_message(&self) -> Message;                // finish-task / session.finish
    fn parse_message(&self, text: &str) -> AsrEvent;    // 现 FunAsrEvent 更名
}
```

现有 `RealtimeAsrFamily::DashscopeDuplex / QwenRealtime` 的分支逻辑分别搬进两个实现（都留在 `providers/alibabacloud/` 内）。`commands/asr.rs` 会话循环持有 `Box<dyn RealtimeAsrConnector>`，删除 5 处 `match family`（连接与首消息 `asr.rs:41-63`、`send_audio_message`、`send_finish`、解析分发 `protocol.rs:180-185`、静音测试）。连接器的选取 = 注册表查协议族 → 供应商模块的工厂函数。

### 3. 多供应商基础设施（第三阶段）

- `normalize_settings` 改为"按**内置 profile 注册函数**补齐缺失项、修正已知内置 profile 的固定字段"，**不再删除未知 profile**；
- `ProviderDefaults` 扩展为多能力结构（`asr: String` 保留 + `#[serde(default)]` 新增字段，或迁移为 map；旧 JSON 必须可读，方案在任务 3.1 内定）；
- `commands/asr.rs` 的 kind 检查改为"注册表/工厂能否给出连接器"；`commands/transcription.rs` 与 `funasr.rs` 的取 Key 改走 `resolve_provider_id` + profile；
- 前端 `ProviderCapability` 改为可扩展类型（含 `"llm"` 预留），`effective()` 兜底、`ensureDictationProviderReady` 改为按当前生效 provider 判断。

### 4. 流程固化与验证（第四阶段）

按手册流程真实新增一个模型（候选 `paraformer-realtime-v2`，DashScope duplex 协议、支持 vocabulary_id，见重要记录 004），验证"只改注册表 + 词表 1 行"即可上线；沉淀《新增供应商与模型操作手册》到 `docs/rules/`。

## 阶段关系与依赖

```
1.1 后端注册表 ──┬── 1.2 前端派生（可与 2.1 并行）
                 └── 2.1 实时连接器 trait ── 3.1 后端解锁 ── 3.2 前端泛化 ── 4.1 验证+手册
```

第一、二阶段是**纯重构**（行为不变），第三阶段涉及持久化结构（需兼容验证），第四阶段是增量验证。

## 兼容性要求

- `say-it-state.json` 旧格式必须无损读入：`defaults` 仅含 `asr` 字段、profiles 仅含 funasr（含 apiKey / hotwords / vocabularyIds）。改动 `ProviderDefaults` 后写一个针对旧 JSON 字符串的反序列化单元测试。
- 前端 Tauri 命令名（`start_asr_stream` / `asr_stream_finish` 等 `ui/src/lib/tauri.ts` 中的 CMD 表）不改名，事件 payload 结构不变。
- 模型下拉的既有选项 value 不变（用户偏好里存的是模型名字符串，`useDictPrefs` / `useSubtitleStore` / `useTranscriptionStore` 各自持久化）。

## 风险与回退

- 主要风险：实时链路重构引入回归（听写收尾时序、字幕自动重连、缓冲音频补发 `protocol.pending`）。控制方式：trait 化时**逐字段搬运不改逻辑**；每任务完成后跑 `cargo check` + `cargo test` + `npx tsc --noEmit` + `npm run ui:build`，并列出人工回归步骤（听写一句注入、字幕开停、录音识别一个文件、热词保存）交用户实测。
- 持久化风险：`normalize_settings` 改动写坏用户配置。控制方式：改动前先为现行为写测试固化，改后测试仍绿；用户实测前手动备份 `say-it-state.json` 的提示写进任务验收。
- 每个任务独立提交，出问题按提交回退。

## 全局开发约束

- 遵守仓库 `CLAUDE.md`：最小影响、复用既有抽象、注释解释"为什么"、每任务独立 Git 提交（简洁中文提交信息）。
- 验证命令：`cargo check`、`cargo test`（在 `src-tauri/` 下）、`npx tsc --noEmit`、`npm run ui:build`；运行验证用 `npm run tauri:dev`（端口固定 5155）。
- 涉及鼠标/快捷键/注入的验证交给用户，任务文件里写清操作步骤与预期。

## 最终验收条件

1. 全部 6 个任务状态为"已完成"；
2. 听写（实时/非实时）、实时字幕、录音识别、文稿对齐、热词管理与改造前行为一致（用户实测确认）；
3. 旧 `say-it-state.json` 升级后 API Key、热词、默认模型偏好全部保留；
4. 第四阶段按手册流程新增的模型真实可用，且改动确实收敛在手册声明的文件范围内；
5. `docs/rules/新增供应商与模型操作手册.md` 存在且与实际代码一致。
